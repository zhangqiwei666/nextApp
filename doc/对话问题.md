## 1. 项目代理端口怎么配置

### 问题
Next.js 项目中如何配置 API 代理端口？

### 回答

通过 `next.config.ts` 的 `rewrites` + 环境变量控制：

```typescript
// next.config.ts
async rewrites() {
  return [
    {
      source: '/api/:path*',
      destination: `${process.env.BACKEND_URL}/api/:path*`,
    },
  ];
},
```

在 `.env.local` 中配置后端地址和端口：

```env
BACKEND_URL=http://115.191.43.57
```

不同环境使用不同 env 文件：

| 文件 | 用途 | 提交 Git |
|------|------|:---:|
| `.env.local` | 本地开发（优先级最高） | ❌ |
| `.env.development` | 开发环境默认值 | ✅ |
| `.env.production` | 生产环境默认值 | ✅ |

修改端口只需修改 `BACKEND_URL` 中的端口号即可。修改后需**重启 dev server**。

## 2. useEffect 中 setState 报错原因

### 问题
`discover-csr/page.tsx` 第 73 行报错：
```
Error: Calling setState synchronously within an effect can trigger cascading renders
```

### 回答

**根本原因**：在 `useEffect` 的执行链路中，`await` **之前**同步调用了多个 `setState`。

```tsx
useEffect(() => {
  fetchData(); // fetchData 内部在 await 之前同步调用了 setState
}, []);

const fetchData = async () => {
  setLoading(true);      // ❌ 同步 setState，触发级联渲染
  setTopics([]);          // ❌ 同步 setState
  setClientTime("...");   // ❌ 同步 setState

  const data = await fetch(...); // await 之后的 setState 没问题
  setTopics(data);        // ✅ 异步回调中，没问题
  setLoading(false);      // ✅ 异步回调中，没问题
};
```

**关键区分**：
- `await` **之前**的 `setState` → effect 的**同步执行阶段** → ❌ 报错
- `await` **之后**的 `setState` → **异步回调** → ✅ 没问题

## 3. 除了 useReducer 还有别的解决方案吗

### 问题
解决 useEffect 中同步 setState 级联渲染，除了 `useReducer`，还有什么方案？

### 回答

| 方案 | 复杂度 | 适合场景 |
|------|:------:|---------|
| **单一 useState 对象** | ⭐ | 状态少、逻辑简单 |
| **useReducer** | ⭐⭐ | 状态多、有复杂状态转换 |
| **startTransition** | ⭐ | 保持原有多个 useState 不改动 |
| **useRef** | ⭐ | 部分状态不需要触发渲染 |

#### 方案 1：单一 useState 对象

```tsx
const [state, setState] = useState({ topics: [], loading: true, ... });
// 一次 setState 更新所有状态
setState(prev => ({ ...prev, loading: true, topics: [] }));
```

#### 方案 2：startTransition

```tsx
import { useTransition } from "react";
const [, startTransition] = useTransition();

startTransition(() => {
  setLoading(true);
  setTopics([]);
});
```

#### 方案 3：useRef 存非渲染数据

```tsx
const fetchDurationRef = useRef(0);
fetchDurationRef.current = duration; // 不触发渲染
```

## 4. 发生的原因是 useCallback 修改状态导致的吗

### 问题
报错是因为 `useCallback` 修改状态，但里面又有同步代码导致的吗？

### 回答

**不完全是，和 `useCallback` 本身无关。** 根本原因是在 `useEffect` 的同步执行阶段调用了 `setState`。

| | 说明 |
|---|---|
| **报错原因** | `useEffect` 同步执行阶段调用了 `setState`（`setLoading`、`setClientTime`） |
| **和 useCallback 的关系** | 无关，换成普通函数或内联函数也一样报错 |
| **和 async 的关系** | `await` 前的代码仍是同步执行的，所以 `await` 前的 `setState` 会报错 |

## 5. request.ts 的 TypeScript 泛型 T 的作用

### 问题
api.ts 里已经传了类型过来，为什么 request.ts 里还需要 `as unknown as T`？

### 回答

`T` 是一个**占位符**，在调用时才确定具体类型：

```
api.ts                          request.ts
──────                          ──────────
http.get<HotTopicsResponse>()  →  request<T>()  // T = HotTopicsResponse
                                   返回 Promise<T>
```

在 `request.ts` 内部，TypeScript 并不知道 `T` 具体是什么。当 `return apiRes` 时：
- 函数承诺返回 `T`（即 `HotTopicsResponse`）
- 但 `apiRes` 的类型是 `ApiResponse<T>`（结构不同）
- TypeScript 无法确认它们相等 → 报错

`as unknown as T` 就是告诉 TypeScript："我知道运行时这个值符合 T，你别管了"。

更简洁的替代方案：直接 `return data as T`，跳过 `ApiResponse` 中间类型。

## 6. 为什么启动项目能访问 /api/discover-crs/list

### 问题
项目里没有对应的 Route Handler，为什么浏览器能访问 `http://localhost:3000/api/discover-crs/list`？

### 回答

因为 `next.config.ts` 的 **`rewrites` 规则**在起作用：

```
浏览器请求: http://localhost:3000/api/discover-crs/list
    ↓
Next.js 发现没有 Route Handler
    ↓
匹配 rewrites 规则: source '/api/:path*'
    ↓
代理到: http://115.191.43.57/api/discover-crs/list
    ↓
外部后端返回数据
```

注意 CSR 和 SSR 走的是**完全不同的路径**：

| | CSR（浏览器） | SSR（Node.js） |
|---|---|---|
| 基础路径 | `''`（相对路径） | `BACKEND_URL` |
| 代理方式 | 通过 `rewrites` 转发 | `request.ts` 直连后端 |

## 7. 客户端为什么拿不到 process.env.BACKEND_URL

### 问题
在 `request.ts` 的 `getBaseURL()` 中，客户端（浏览器）取 `process.env.BACKEND_URL` 是 `undefined`。

### 回答

**Next.js 安全机制**：只有 `NEXT_PUBLIC_` 开头的环境变量才会暴露给浏览器。

```env
# ❌ 浏览器拿不到
BACKEND_URL=http://115.191.43.57

# ✅ 浏览器能拿到
NEXT_PUBLIC_BACKEND_URL=http://115.191.43.57
```

构建时 Next.js 对客户端代码做的事：

```javascript
// NEXT_PUBLIC_ 开头 → 值被写死进代码
process.env.NEXT_PUBLIC_API_URL  →  "http://xxx"  ✅

// 非 NEXT_PUBLIC_ → 不替换，浏览器里 undefined
process.env.BACKEND_URL  →  undefined  ❌
```

**但 CSR 不需要直连后端**，用相对路径走 `rewrites` 代理即可，所以 `getBaseURL()` 客户端返回 `''` 是正确的。

## 8. .env.local 和 .env.development 的区别

### 问题
`.env.local` 和 `.env.development` 有什么区别？

### 回答

#### 加载优先级（高 → 低）

```
.env.local          ← 最高优先级
.env.development    ← 仅 dev 环境
.env.production     ← 仅 production 环境
.env                ← 最低，所有环境通用
```

同名变量以高优先级文件为准。

#### 用途对比

| | `.env.local` | `.env.development` |
|---|---|---|
| **用途** | 本机私有配置 | 开发环境通用默认值 |
| **提交 Git** | ❌ 不提交 | ✅ 提交 |
| **谁用** | 只有你自己 | 团队所有人 |

## 9. 开发环境 rewrites 代理到 localhost:3000 怎么请求到接口的

### 问题
`next.config.ts` 中 rewrites 的默认值是 `http://localhost:3000`，开发环境怎么请求到后端接口的？

### 回答

**实际并没有代理到 `localhost:3000`**。

```typescript
destination: `${process.env.BACKEND_URL || 'http://localhost:3000'}/api/:path*`
```

因为 `.env.local` 设置了 `BACKEND_URL=http://115.191.43.57`，`||` 表达式取的是 `BACKEND_URL` 的值。`http://localhost:3000` 只是兜底默认值。

如果真的回落到 `localhost:3000`，会出现**死循环**（请求自己 → 又触发 rewrite → 又请求自己）。

建议把默认值去掉，避免误用：

```typescript
destination: `${process.env.BACKEND_URL}/api/:path*`,
```

## 10. next.config.ts 能取到 env 但 request.ts 客户端取不到

### 问题
`next.config.ts` 能取到 `process.env.BACKEND_URL`，但 `request.ts` 在客户端取不到，为什么？

### 回答

因为它们运行在**不同的环境**：

| 代码位置 | 运行环境 | 能访问 `BACKEND_URL`？ | 原因 |
|---------|---------|:---:|------|
| `next.config.ts` | Node.js（启动时） | ✅ | 纯服务端代码，只在 `yarn dev` 时执行 |
| `request.ts` SSR | Node.js（请求时） | ✅ | 服务端执行 |
| `request.ts` CSR | 浏览器 | ❌ | 非 `NEXT_PUBLIC_` 前缀，不注入浏览器 |

`next.config.ts` 在 Node.js 启动时执行一次，配好 rewrites 规则后就不再运行了。而 `request.ts` 会被打包进浏览器 JS，浏览器里没有 `process.env`。

这也是 `getBaseURL()` 判断 `window` 的意义：

```
               同一份 request.ts 代码
              ┌─────────────────────┐
              │   getBaseURL()      │
              └──────┬──────────────┘
                     │
        ┌────────────┴────────────┐
        ▼                         ▼
   浏览器（CSR）              Node.js（SSR）
   返回 ''                   返回 BACKEND_URL
        │                         │
        ▼                         ▼
   走 rewrites 代理           直连后端
```

**客户端不该直连后端**，用相对路径走 `rewrites` 由 Next.js 服务端转发。
